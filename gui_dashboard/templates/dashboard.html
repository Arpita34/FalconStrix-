      Process Creation & Management

      Implemented using fork() in main.cpp. Parent supervises the system while child runs the monitoring engine. Uses waitpid() for zombie process handling.

      After fork, parent and child are separate processes: different memory, same program text until they diverge.

      fork()wait()PCBParent-Child

      Multithreading (Concurrency)

      Uses POSIX threads for parallel execution: process monitoring, behavioral detection, resource monitoring — all running concurrently.

      Threads are lighter than processes: same PID, shared globals, but independent call stacks and instruction pointers.

      pthread_createpthread_joinParallel

      Synchronization (Mutex)

      Shared data protected using pthread_mutex_t. Both monitor and detector threads acquire the lock before reading /proc.

      Without the lock, two readers could see a PID list change mid-scan—mutexes turn that into “one at a time.”

      MutexCritical SectionThread Safety

      Inter-Process Communication

      Named Pipes (FIFO) connect Red Team and C++ Engine to the Python Backend. Non-blocking write on C++ side, blocking read on Python side.

      Think of it as a shared file or FIFO buffer: writers push bytes; the backend drains lines and parses JSON.

      Named PipesmkfifoO_NONBLOCK

      Signal Handling

      Registers signal(SIGINT, handler) and signal(SIGTERM, handler) for graceful shutdown. Parent forwards signals to child.

      Signals are OS-level interrupts to a process; handlers run asynchronously and should only flip safe flags and forward work.

      SIGINTSIGTERMAsync Events

      Process Control & Termination

      Suspicious processes terminated using kill(pid, SIGKILL) in both C++ and Python. OS-level enforcement.

      SIGKILL cannot be caught—use only when you intend to tear the target down immediately.

      kill()SIGKILLos.kill()

      /proc Filesystem Interaction

      Reads /proc/[pid]/comm, /proc/[pid]/cmdline, and /proc/[pid]/status for process name, command, state, threads, memory.

      /proc is not a normal disk tree—it is the kernel exposing live process and system state as text files.

      Dedicated thread reads /proc/stat for CPU and /proc/meminfo for memory. Alerts at >90% threshold.

      Process Creation & Management

      fork(), wait(), Parent-Child Architecture

      How It Works (from real main.cpp)

      Big picture: main() runs a while (running) loop. Each iteration calls fork() once. The child only runs the monitoring engine (run_child_engine() + _exit(0)). The parent saves the child PID, then blocks in waitpid() until the child exits. If the child crashes or is killed, the parent waits 2 seconds and forks again (fault recovery).

      Why it looks odd at first: after fork(), the same source file keeps running in two processes. The return value of fork() tells you which role you are in—pid == 0 means child, pid > 0 means parent (that number is the child’s PID). The parent must waitpid so the kernel can free the child’s bookkeeping entry; otherwise you get zombie processes stacking up.

      Use the right panel for the live diagram; the source code below highlights the matching block and scrolls to it. Same six steps — 5 seconds each, 30 seconds per full cycle. Step ⑤ uses WIFEXITED (normal exit code) vs WIFSIGNALED (killed by signal) to decide whether to restart.

// 5. Signal Handling — async-signal-safe flag
volatile sig_atomic_t running = 1;
static pid_t child_pid = -1;

// 5. Parent signal handler
void signal_handler(int signum) {
    if (signum == SIGINT) {
        std::cout << "\n[SIGNAL] Caught SIGINT (Ctrl+C), initiating graceful shutdown..." << std::endl;
    } else if (signum == SIGTERM) {
        std::cout << "\n[SIGNAL] Caught SIGTERM, initiating graceful shutdown..." << std::endl;
    }
    running = 0;
    // Forward signal to child for coordinated shutdown
    if (child_pid > 0) {
        kill(child_pid, signum);
    }
}

// 5. Child signal handler
void child_signal_handler(int signum) {
    std::cout << "[SIGNAL] Child received signal " << signum << ", stopping threads..." << std::endl;
    running = 0;
}

// Child process work
void run_child_engine() {
    signal(SIGINT, child_signal_handler);
    signal(SIGTERM, child_signal_handler);

    pthread_t monitor_thread, detector_thread, resource_thread, response_thread;
    void* flag_ptr = (void*)&running;

    pthread_create(&monitor_thread, NULL, start_process_monitor, flag_ptr);
    pthread_create(&detector_thread, NULL, start_behavior_detector, flag_ptr);
    pthread_create(&resource_thread, NULL, start_resource_monitor, flag_ptr);
    pthread_create(&response_thread, NULL, start_response_engine, flag_ptr);

    pthread_join(monitor_thread, NULL);
    pthread_join(detector_thread, NULL);
    pthread_join(resource_thread, NULL);
    pthread_join(response_thread, NULL);

    std::cout << "[*] Child engine: All worker threads stopped." << std::endl;
}

int main() {
    std::cout << "[*] Starting Blue Team OS Engine..." << std::endl;
    signal(SIGINT, signal_handler);
    signal(SIGTERM, signal_handler);

    while (running) {
        pid_t pid = fork();
        if (pid < 0) {
            std::cerr << "[ERROR] Fork failed." << std::endl;
            return 1;
        }
        if (pid == 0) {
            run_child_engine();
            _exit(0);
        } else {
            child_pid = pid;
            std::cout << "[*] Parent Engine supervising child workers (PID: " << pid << ")" << std::endl;
            int status;
            waitpid(pid, &status, 0);

            if (!running) break;
            if (WIFEXITED(status)) {
                int exit_code = WEXITSTATUS(status);
                if (exit_code != 0) {
                    std::cerr << "[!] Child exited with error code " << exit_code
                              << ", restarting in 2 seconds..." << std::endl;
                    sleep(2);
                    continue;
                } else {
                    std::cout << "[*] Child exited normally (code 0)." << std::endl;
                    break;
                }
            } else if (WIFSIGNALED(status)) {
                int term_sig = WTERMSIG(status);
                std::cerr << "[!] Child killed by signal " << term_sig
                          << ", restarting in 2 seconds..." << std::endl;
                sleep(2);
                continue;
            }
        }
    }
    std::cout << "[*] Blue Team OS Engine shut down cleanly." << std::endl;
    return 0;
}

        Multithreading (Concurrency)

        POSIX Threads — pthread_create, pthread_join

        How It Works

        After fork(), the child installs four workers via pthread_create; here we focus on three that run until running becomes 0. They share the same address space (global mutex, same PID) but each has its own stack and program counter. The OS scheduler interleaves them preemptively—you are not guaranteed strict round-robin order. Shorter sleep(2) on the behavior detector means it can wake more often than the monitors (5s), so it can spot spikes or suspicious processes between slower ticks. Blocking in sleep yields the CPU to siblings without busy-waiting.

        Remember: getpid() is the same for all three threads—they are not separate processes. pthread_create only starts execution; pthread_join (in run_child_engine) waits for each worker to finish when shutting down. If two threads touched the same unsynchronized data, you would risk races; here shared /proc reads go through the mutex helpers you see on the Synchronization page.

        Synchronization (Mutex)

        Critical Section Protection, Race Condition Prevention

        How It Works

        get_all_pids() walks /proc with opendir/readdir: the kernel can change directory contents while you read, so two threads doing that without coordination could see torn or inconsistent snapshots. A single pthread_mutex_t proc_mut makes get_process_count() and get_all_pids_safe() critical sections: lock → read → unlock. Only one thread holds the mutex at a time; the other blocks in pthread_mutex_lock (not spinning forever in this API). The raw helper stays unguarded so it can be called only from already-locked code paths—never expose it to concurrent callers without the same lock order everywhere (avoid deadlock by always taking proc_mut alone).

        Analogy: the mutex is a single bathroom key—only one thread “inside” /proc at a time. get_all_pids_safe copies the vector while holding the lock, then returns the copy so the detector can scan PIDs without holding the lock for the whole loop.

        Source code — process_monitor.cpp & behavior_detector.cpp

        The highlighted block follows the mutex story on the right (raw enumerate → monitor count → safe copy → detector use), 5s per step. The panel scrolls so the lock state and the code stay aligned.

        // Global mutex — shared between all threads
pthread_mutex_t proc_mut = PTHREAD_MUTEX_INITIALIZER;

// Raw /proc enumeration (NOT thread-safe)
std::vector<int> get_all_pids() {
    std::vector<int> pids;
    DIR* dir = opendir("/proc");
    if (!dir) return pids;

    struct dirent* entry;
    while ((entry = readdir(dir)) != NULL) {
        if (entry->d_type == DT_DIR) {
            std::string name = entry->d_name;
            bool is_pid = true;
            for (char c : name) {
                if (!isdigit(c)) { is_pid = false; break; }
            }
            if (is_pid && !name.empty()) {
                pids.push_back(std::stoi(name));
            }
        }
    }
    closedir(dir);
    return pids;
}
// Mutex-protected process count
int get_process_count() {
    pthread_mutex_lock(&proc_mut);
    int count = get_all_pids().size();
    pthread_mutex_unlock(&proc_mut);
    return count;
}
// Mutex-protected PID list access (thread-safe)
std::vector<int> get_all_pids_safe() {
    pthread_mutex_lock(&proc_mut);
    std::vector<int> pids = get_all_pids();
    pthread_mutex_unlock(&proc_mut);
    return pids;
}
// behavior_detector.cpp — mutex-protected snapshot

for (int p : pids) {
    std::string pname = get_process_name(p);

        send_event_to_backend("SUSPICIOUS_PROC",

        Inter-Process Communication

        Named Pipes (FIFO), Message Passing

        How It Works

        All producers agree on one path from ipc_config.py so C++, Python, and Red Team scripts never write to different files. Events are newline-delimited JSON (one object per line) so the reader can call readline() and parse each payload independently. The C++ side opens with O_NONBLOCK so a slow consumer cannot block the detector thread indefinitely on open/write; if the pipe is not ready, the write fails fast and the engine returns. The backend typically opens for read in a long-lived loop; Red Team tools often append ('a') like a log. On Windows the “pipe” may be a regular file the backend creates; on Linux a true FIFO (mkfifo) is common.

        Failure modes: if the backend is not running, the path may not exist yet—start main_backend.py first. Partial writes or invalid JSON lines will break parsing; keeping one JSON object per line avoids needing a streaming parser.

        Signal Handling

        SIGINT, SIGTERM, Graceful Shutdown

        How It Works

        Signal handlers run in a restricted context: only async-signal-safe operations are guaranteed. A plain bool flag might be torn if read/written from a signal and a thread at once; volatile sig_atomic_t is the portable way to store a small “please stop” flag. The parent records child_pid after fork so signal_handler can kill(child_pid, signum)—forwarding the same signal lets the child run its own handler and clear running so pthread worker loops exit. Without that forward, the child might keep running after the parent decides to shut down. SIGINT (Ctrl+C) and SIGTERM are both hooked to the same handler for consistent cleanup.

        Contrast: the default for many signals is to terminate the process immediately. Installing a handler lets you set flags and notify children first. Avoid heavy work (heap allocation, locks) inside the handler—set running=0 and let normal thread code finish the job.

        Resource Monitoring

        CPU & Memory Tracking via /proc

        How It Works

        A dedicated resource monitor thread (see start_resource_monitor) wakes every 5 seconds, samples the kernel pseudo-filesystems, and logs CPU and memory percentages. CPU % is not read directly from the kernel: the code stores the previous jiffies from /proc/stat, compares the next sample, and turns Δ busy time / Δ total time into usage_percent. Memory % prefers MemAvailable (what the kernel thinks apps can allocate) over raw MemFree, then derives used memory vs MemTotal. ProcessStats and get_process_stats(pid) mirror optional per-process fields from /proc/[pid]/stat (utime, stime, RSS, thread count). When either percentage exceeds 90%, the engine emits HIGH_CPU or HIGH_MEMORY through the same IPC path as other detectors.

        Reading the numbers: the first get_cpu_usage() call only primes prev_*—the second call produces a meaningful percentage. The dashboard gauges on the right are simulated for teaching clarity; on a real Linux host the same structs would reflect live /proc parses.

#include <iostream>
#include <unistd.h>

static unsigned long long prev_total = 0;
static unsigned long long prev_idle = 0;

// Read CPU jiffies from /proc/stat (first line)
CpuUsage get_cpu_usage() {
    CpuUsage cpu = {0, 0, 0, 0, 0};
    std::ifstream stat_file("/proc/stat");
    if (!stat_file.is_open()) return cpu;

    std::string line;
    std::getline(stat_file, line);
    std::istringstream iss(line);
    std::string cpu_label;
    unsigned long long user, nice, system, idle, iowait, irq, softirq, steal;
    iss >> cpu_label >> user >> nice >> system >> idle
        >> iowait >> irq >> softirq >> steal;

    cpu.user = user; cpu.nice = nice;
    cpu.system = system; cpu.idle = idle;
    unsigned long long total_idle = idle + iowait;
    unsigned long long total = user + nice + system + idle
        + iowait + irq + softirq + steal;

    unsigned long long diff_idle = total_idle - prev_idle;
    unsigned long long diff_total = total - prev_total;
    if (diff_total > 0) {
        cpu.usage_percent = (double)(diff_total - diff_idle)
            / (double)diff_total * 100.0;
    }
    prev_idle = total_idle;
    prev_total = total;
    return cpu;
}
// Read memory keys from /proc/meminfo
MemoryInfo get_memory_info() {
    MemoryInfo mem = {0, 0, 0, 0, 0, 0.0};
    std::ifstream meminfo("/proc/meminfo");
    if (!meminfo.is_open()) return mem;

    std::string line;
    while (std::getline(meminfo, line)) {
        std::istringstream iss(line);
        std::string key;
        unsigned long value;
        iss >> key >> value;
        if (key == "MemTotal:")          mem.total_kb = value;
        else if (key == "MemFree:")      mem.free_kb = value;
        else if (key == "MemAvailable:") mem.available_kb = value;
        else if (key == "Buffers:")      mem.buffers_kb = value;
        else if (key == "Cached:")       mem.cached_kb = value;
    }
    if (mem.total_kb > 0) {
        unsigned long used = mem.total_kb - mem.available_kb;
        mem.usage_percent = (double)used / (double)mem.total_kb * 100.0;
    }
    return mem;
}
// Resource monitor thread — sample every 5 seconds
void* start_resource_monitor(void* arg) {
    std::cout << "[+] Resource Monitor thread started." << std::endl;
    get_cpu_usage();  // Prime the delta calculation
    sleep(1);

    while (true) {
        CpuUsage cpu = get_cpu_usage();
        MemoryInfo mem = get_memory_info();

        std::cout << "[RES] CPU: " << cpu.usage_percent
                  << "% | MEM: " << mem.usage_percent << "%"
                  << " (" << (mem.total_kb - mem.available_kb) / 1024
                  << "MB / " << mem.total_kb / 1024 << "MB)"
                  << std::endl;
        if (cpu.usage_percent > 90.0) {
            send_event_to_backend("HIGH_CPU",
                "CPU usage exceeded 90% threshold",
                getpid(), "resource_monitor", 3);
        }
        if (mem.usage_percent > 90.0) {
            send_event_to_backend("HIGH_MEMORY",
                "Memory usage exceeded 90% threshold",
                getpid(), "resource_monitor", 3);
        }
        sleep(5);
    }
    return NULL;
}

        def active_defense():
    """Invoked when FSM State -> LOCKED.
    Kills all processes linked to unresolved HIGH/CRITICAL alerts."""
    query = """
    SELECT DISTINCT p.pid, p.process_name, e.event_id
    FROM Processes p
    JOIN Events e ON p.process_id = e.process_id
    JOIN Alerts a ON e.event_id = a.event_id
    JOIN Severity s ON a.severity_id = s.severity_id
    WHERE a.is_resolved = FALSE
      AND s.level_name IN ('HIGH', 'CRITICAL', 'MEDIUM')
    """
    bad_procs = fetch_query(query)

    for proc in bad_procs:
        pid = proc['pid']
        name = proc['process_name']
        try:
            os.kill(pid, signal.SIGKILL)
            msg = f"Terminated malicious process {name} (PID: {pid})"
            print(f"[*] RESPONSE ENGINE: {msg}")
            execute_query(
                "INSERT INTO Events (event_type, description, source) "
                "VALUES (%s, %s, %s)",
                ('RESPONSE_ACTION', msg, 'Response_Engine'))
            execute_query(
                "UPDATE Alerts SET is_resolved=TRUE WHERE event_id=%s",
                (proc['event_id'],))
        except ProcessLookupError:
            pass
        except PermissionError:
            print(f"[!] Permission denied to kill PID {pid}")
